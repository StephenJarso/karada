# Karada (🎴) — Trustless Cross-Border Lightning Escrow

**Karada** (from the Swahili concept of trust and credit) is a trustless, peer-to-peer international e-commerce escrow protocol built natively on Bitcoin's Layer 2 Lightning Network. It eliminates the "cross-border trust deficit" between global buyers (e.g., in the USA) and merchants in emerging markets (e.g., in Kenya) without relying on traditional banks, expensive credit card rails, or third-party escrow services.

---

## 🚀 The Core Problem & Solution

### The Friction in Global Trade
When an artisan in Nairobi sells high-value goods to a buyer in New York, a massive trust gap exists:
* **The Buyer's Risk:** Paying upfront introduces the risk that the merchant disappears or ships sub-par cargo.
* **The Merchant's Risk:** Shipping first exposes the merchant to payment default or malicious credit card chargebacks ("friendly fraud"), where traditional payment gateways uniformly favor Western buyers and pass penalty fees back to African small businesses.
* **The Legacy Fiat Wall:** Traditional escrow providers charge excessive commissions (3%–6%), demand high international wire minimums, and tie up capital for 3 to 7 business days.

### The Karada Engine
Karada replaces centralized multi-million-dollar banking intermediaries with native **Bitcoin Script Primitives**—specifically **Hash Time-Locked Contracts (HTLCs)** and **HODL Invoices**. 
* Capital is locked cryptographically *in flight* inside the Lightning Network.
* **No Chargebacks:** Once the transaction settles, payments are final and mathematically irreversible.
* **Near-Zero Fees:** Settlement happens instantly for pennies (satoshis), allowing merchants to retain 99.9% of their margins.

## 🛠️ Cryptographic Architecture & Flow

Karada coordinates trust by acting as a **Data Oracle** that manages the disclosure of a cryptographic secret (the `preimage`).

### The Protocol Sequence

```
[ US Customer ] ----(1) Routes Held Payment ----> [ Lightning Network Nodes ]
|                                                      |
(3) Ships Cargo via DHL                             (5) Claims Sats via Preimage
v                                                      v
[ Nairobi Merchant ] <---(2) Invoices Held Locked---- [ Karada Go Engine (Oracle) ]
|                                                      ^
+-------------(4) Submits Tracking ID -----------------+
```

1. **Commitment:** The US Customer commits funds to a specific `Payment Hash` generated by Karada. The funds leave their wallet but are held securely along the Lightning route.
2. **Notification:** The Karada Go engine detects that the HTLC is locked (`ACCEPTED`). It alerts the Nairobi Merchant that the funds are secured by math.
3. **Dispatched:** The merchant ships the item via an international carrier (e.g., DHL) and submits the tracking number to Karada.
4. **Settlement (Success Path):** Karada's automated backend polls the carrier's API. The split-second the shipment is registered as "Delivered," Karada releases the 32-byte `preimage` to the merchant's Lightning node. The merchant claims the funds instantly.
5. **Protection (Refund Path):** If the merchant fails to log a valid shipment within a predefined Bitcoin time-lock window (e.g., 432 blocks / ~3 days), the HTLC expires, and the network automatically routes the locked funds back to the customer's wallet.

## 🏗️ Technical Stack & Project Structure

The platform is engineered using a clean, domain-driven architecture optimized for ultra-low latency and zero-state thread safety.

* **Backend Engine:** Go (Golang)
* **Lightning Interface:** gRPC communication layer to `lnd` (Lightning Network Daemon)
* **Datastore:** In-Memory Cache (optimized for 48-hour hackathon execution)

```text
karada/
├── cmd/
│   └── server/
│       └── main.go          # HTTP/gRPC API entry point
├── internal/
│   ├── escrow/
│   │   ├── handler.go       # REST Controllers for checkout & settlement
│   │   ├── service.go       # Core business logic & state transition engines
│   │   └── repository.go    # Safe concurrent datastore memory access
│   ├── lightning/
│   │   └── client.go        # LND interaction client (AddHoldInvoice, SettleInvoice)
│   └── oracle/
│       └── courier.go       # Webhook receiver / Courier tracking simulator
└── pkg/
    └── crypto/
        └── random.go        # High-entropy 32-byte preimage generator
```

## 🚦 Quick Start & Local Setup

### Prerequisites

* Go 1.21 or higher
* A functional local Lightning Developer environment (Polar, Regtest, or Simnet with lnd)

### 1. Clone and Install Dependencies

```bash
git clone https://github.com/your-username/karada.git
cd karada
go mod tidy
```

### 2. Configure LND Credentials

Export your local node authentication data so the Go engine can interact with the gRPC interface:

```bash
export LND_RPC_HOST="127.0.0.1:10009"
export LND_MACAROON_PATH="/path/to/admin.macaroon"
export LND_TLS_CERT_PATH="/path/to/tls.cert"
```

### 3. Run the Server

```bash
go run cmd/server/main.go
```

The server will bind to http://localhost:8080.

## 🧪 Interactive Hackathon Simulation Script

To present Karada to hackathon judges without deploying live intercontinental cargo, use the built-in HTTP simulator endpoints to demonstrate both paths:

### Step 1: Create a HODL Escrow Invoice

The buyer decides to purchase an item worth 50,000 Satoshis.

```bash
curl -X POST http://localhost:8080/api/v1/escrow \
  -H "Content-Type: application/json" \
  -d '{"amount_sats": 50000, "description": "Hand-carved Mahogany Art"}'
```

Response: Returns a standard Lightning Payment Request string (lnbc...) along with a tracked payment_hash.

### Step 2: Simulate Payment Hold

The judge pays the invoice using a test wallet. The funds lock. Karada updates the tracking state:

```bash
# Backend internally catches the payment update via LND invoice subscription
# Status shifts to HELD
```

### Step 3: Merchant Logs Shipment

The merchant drops the art off at the courier hub and updates Karada:

```bash
curl -X POST http://localhost:8080/api/v1/escrow/ship \
  -H "Content-Type: application/json" \
  -d '{"payment_hash": "<HASH>", "tracking_number": "DHL-NBO-9921"}'
```

### Step 4: Trigger the Oracle Delivery Simulator (The Payoff)

Simulate the physical arrival of the package in New York via the Oracle interceptor webhook:

```bash
curl -X POST http://localhost:8080/api/v1/oracle/simulate-delivery \
  -H "Content-Type: application/json" \
  -d '{"tracking_number": "DHL-NBO-9921"}'
```

Result: Karada instantly evaluates the proof, provides the secret preimage to the merchant node, and the merchant's wallet balance updates on screen in real-time.
